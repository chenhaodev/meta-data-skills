## Dataset Overview

**HealthBench** is an open-ended medical AI evaluation benchmark created by **OpenAI** (2025).
It consists of **5,000 multi-turn conversations** graded against **48,562 unique rubric criteria**
written by **262 physicians**. All content is in English. Tasks are open-generation (not MCQ):
a model must produce a free-text health response that is then scored against physician-written rubrics.

Paper: *HealthBench: A Benchmark for Evaluating Large Language Models in Health* (arXiv 2505.08775)

## Clinical Domain & AI Relevance

**Health contexts covered:**
- Emergency triage and urgent care guidance
- Transforming/interpreting clinical data (labs, vitals, imaging reports)
- Global health and resource-constrained settings
- General patient communication across multiple specialties

**Behavioral axes assessed:**
- `axis:accuracy` — factual correctness
- `axis:completeness` — coverage of clinically relevant points
- `axis:context_awareness` — adapting to patient/provider context
- `axis:communication_quality` — clarity, empathy, appropriate hedging

**Why it matters:** Unlike MCQ benchmarks that are saturated (GPT-4-class models score 80–90% on MedQA),
HealthBench uses open-ended rubric grading that still differentiates frontier models.

**SOTA scores (rubric % correct):**
| Model | Score |
|-------|-------|
| GPT-3.5 Turbo | 16% |
| GPT-4o | 32% |
| o3 | 60% |
| GPT-4.1 nano | ~25% (beats GPT-4o at 25× lower cost) |

## Key Splits & Subsets

| Split | Size | Notes |
|-------|------|-------|
| test (default) | 5,000 conversations | Primary eval split |
| HealthBench Hard | subset | More challenging subset for frontier model differentiation |
| HealthBench Consensus | subset | 34 physician-validated dimensions; highest agreement |

Key rubric tags (queryable via `example_tags` field):
- `theme:emergency_referrals`, `theme:context_seeking`, `theme:hedging`
- `level:example` vs `level:cluster` (item-level vs group-level scoring)

## Recommended Use Cases

- **Benchmarking LLMs:** Run `run_benchmark()` with your model callable; compare `mean_score` against
  the SOTA table above. Use HealthBench Hard subset for frontier model differentiation.
- **Fine-tuning data selection:** Filter for `axis:accuracy` rubrics with high physician agreement
  (HealthBench Consensus) to select training examples with clear ground truth.
- **Few-shot prompt selection:** Query by `theme:` tag matching your deployment context
  (e.g., emergency, global health) to find representative in-context examples.
- **Agentic evaluation:** Feed multi-turn `prompt` conversation history to agentic systems;
  rubric grading naturally handles multi-step responses.

## Pitfalls & Gotchas

1. **Schema inconsistency:** The HuggingFace dataset viewer is broken due to column mismatch between
   JSONL files (`rubrics` vs `binary_labels`/`rubric`). Load raw JSONL files directly rather than
   using `datasets.load_dataset()` with auto-schema detection — or pin to a specific file.
2. **Not an MCQ benchmark:** There is no single correct answer string. `score()` requires rubric
   grading (ideally LLM-as-judge against each criterion). The stub implementation uses substring
   match against `ideal_completion` as a proxy only.
3. **Physician rubric subjectivity:** Even with 262 physicians, inter-rater agreement varies by
   theme. `HealthBench Consensus` is the safer subset if you want higher-confidence ground truth.
4. **Contamination risk:** Released May 2025; any model trained after that date may have seen the
   rubrics. Use `canary` field to detect if a model has memorized prompts verbatim.
5. **Rubric scoring cost:** Full rubric evaluation (48,562 criteria) requires LLM-as-judge calls
   per rubric item — expensive at scale. Use `sample()` with small `n` for quick iteration.
