# NEJM Clinical Annotations Dataset Expert

> This skill was generated by `/meddata:build nejm`.

## Dataset Overview

**NEJM Clinical Annotations** is a human expert annotation dataset published in
*Nature npj Digital Medicine* (2024). It contains **101 NEJM clinical vignettes**
(USMLE-style open-ended cases) answered by an LLM, each independently reviewed by
**two physician experts** who assessed correctness and counted errors in the model's
reasoning.

- **101 valid examples** (104 rows total; 3 are metadata/summary rows)
- **11 columns:** question, ground-truth answer, prompt style, model answer, model rationale,
  binary correctness label, two sets of reviewer error counts and explanations
- **Language:** English
- **Modality:** text (clinical vignettes + model responses + expert annotations)

Paper: *Nature npj Digital Medicine* 7, 103 (2024).
DOI: https://doi.org/10.1038/s41746-024-01010-1
Supplementary Data 5 = `41746_2024_1010_MOESM9_ESM.xlsx`

## Clinical Domain & AI Relevance

**Clinical domains covered:** Internal medicine, emergency medicine, cardiology, neurology,
nephrology, endocrinology, and other specialties typical of NEJM clinical case records.
Cases range from acute presentations (altered mental status, AKI) to chronic disease management.

**Three task framings this dataset enables:**

1. **Open-ended clinical QA (benchmarking):** Use `question` + `answer` to test any new model's
   accuracy on physician-level NEJM cases. Ground-truth answers are short clinical phrases
   (e.g., "Cholesterol embolization").

2. **NLI / error classification:** Given question + model answer + rationale, classify as
   correct (Y) or incorrect (N). Useful for training/evaluating medical fact-checking models.

3. **Expert comparison:** Analyse inter-rater agreement between Reviewer 1 and Reviewer 2
   on error counts — both agreement and divergence are clinically informative signals.

**Observed model performance (embedded in the dataset — all 'DR' prompt style):**
- Correct (Y): 89/101 = **88.1%** by ground truth label
- Zero errors by Reviewer 1: 81/103 = **78.6%**
- Zero errors by Reviewer 2: 89/104 = **85.6%**
- Inter-reviewer divergence: ~7–10% of cases rated differently by the two experts

## Recommended Use Cases

- **Benchmarking LLMs:** Feed `question` to your model; compare output against `answer` via
  exact match (for short answers) or substring match. The 101-case set is small but clinically
  rich; complement with MedQA or HealthBench for scale.
- **Fine-tuning data selection:** Filter to `correct_clean == "Y"` + `r1_errors == 0`
  + `r2_errors == 0` (~75 cases) for high-confidence correct reasoning examples.
  Use `rationale` + `answer` as target for SFT on clinical CoT.
- **Few-shot prompt selection:** Sample 3–5 cases where both reviewers agreed the answer
  was correct (zero errors) to use as demonstrative in-context examples.
- **Agentic evaluation / NLI:** Feed the existing `model_answer` + `rationale` to an LLM
  judge; compare its correctness rating against the two human expert labels. Measures
  LLM-as-judge calibration on clinical text.

## Pitfalls & Gotchas

1. **Dirty `Correct?` column:** Values include 'Y', 'N', 'f', 'false arguments', and
   integers (18, 20). Always normalise: treat 'Y' as correct, everything else as incorrect
   or drop. Use `_parse_correct()` in `pipeline.py`.
2. **Fractional error counts:** Reviewer columns contain floats like `0.818` and `0.113` in
   ~2 rows — these appear to be ratios or data entry errors, not integer error counts.
   Treat values > 3 or non-integer as NaN when computing statistics.
3. **3 non-data rows:** The Excel file has 104 rows but 3 have `NaN` index — they are
   summary/metadata rows. `load_dataset()` drops them automatically.
4. **Small size:** 101 examples is too small for reliable statistical significance on its own.
   Report confidence intervals or combine with other NEJM-style benchmarks.
5. **Contamination risk:** NEJM clinical cases are publicly available; any model trained on
   web data after 2024 may have seen these questions. Results should be interpreted alongside
   newer held-out benchmarks.
